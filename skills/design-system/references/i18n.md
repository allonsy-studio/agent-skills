# Internationalization (i18n) for Design System Components

Your components will be consumed by teams serving audiences you didn't
plan for. A card component built for an English-language app will
eventually render Arabic product descriptions. A date picker designed for
MM/DD/YYYY will meet a German team expecting DD.MM.YYYY. A button that
fits perfectly at 8 characters will face a 25-character Finnish translation.

This reference covers the i18n concerns that design system engineers need
to build into components from the start — not bolt on later.

## Text direction and layout

### RTL is a layout concern, not a text concern

When the `dir` attribute is `rtl`, the entire layout mirrors:
- Flex row and grid column directions reverse
- Text-align defaults flip
- Scroll direction reverses
- Position-based layout (left/right) must flip

But the component shouldn't need to know about this if it's built with
logical properties (see next section).

### The dir attribute

Components should respect the document's direction without requiring
configuration:

```html
<html dir="rtl" lang="ar">
```

If a component contains content in a different direction from the page
(e.g., an LTR code snippet inside an RTL page), use the `dir` attribute
locally:

```html
<div dir="rtl" lang="ar">
  <p>هذا هو النص العربي</p>
  <pre dir="ltr"><code>const x = 42;</code></pre>
</div>
```

### Bidirectional text (bidi)

Mixed-direction text is handled by the Unicode Bidirectional Algorithm,
but components can help:

- Use `<bdi>` (bidirectional isolate) to wrap user-generated content that
  might be in a different direction from its context:

```html
<p>User <bdi>محمد</bdi> left a comment</p>
```

Without `<bdi>`, the Arabic name can disrupt the English sentence's
punctuation and layout.

- Use `unicode-bidi: isolate` in CSS for the same effect on styled elements.

## Logical properties

### The heuristic

**Use logical properties when the value should follow the reader. Use
physical properties when the value should follow the screen.**

Logical properties (`margin-inline-start`, `padding-block-end`,
`inset-inline-end`) map to the flow of content — the direction text runs
(inline axis) and the direction text stacks (block axis). When you write
`margin-inline-start`, you're making a semantic claim: "this margin belongs
on the side where text begins." In LTR English, that's the left. In RTL
Arabic, that's the right. In vertical Japanese, it's the top.

That claim should be true. Using logical properties where the value doesn't
actually follow content flow obscures intent and can introduce bugs — a
viewport-anchored FAB using `inset-inline-end: 24px` will jump to the
wrong side of the screen in RTL, because you told it to follow the reader
when you meant it to follow the screen.

### Where logical properties are the right call

Anything tied to content, text flow, or content containment:

- **Text spacing:** `padding-inline`, `padding-block`,
  `margin-block-end` between paragraphs
- **Content containers:** cards, articles, prose blocks, form fields
- **Borders that frame content:** `border-inline-start` for a blockquote
  accent stripe, `border-block-end` for a section divider
- **Lists, tables, anything where reading order matters**
- **Component internals where the component holds content**

The test: if this site were translated to Arabic or Japanese tomorrow,
would I want this value to flip with the language? If yes → logical.

Example — an alert with a severity stripe:
```css
/* Physical: won't flip in RTL — wrong if the alert contains text content */
.ds-alert {
  border-left: 4px solid var(--ds-alert-accent-color);
  padding-left: 1rem;
}

/* Logical: flips automatically — correct, the stripe follows reading flow */
.ds-alert {
  border-inline-start: 4px solid var(--ds-alert-accent-color);
  padding-inline-start: 1rem;
}
```

### Where physical properties are still correct

Anything tied to the viewport, device, or physical layout rather than
content flow:

1. **Viewport-relative positioning.** A "back to top" button anchored to
   `bottom: 24px; right: 24px` should stay in the bottom-right corner
   regardless of language. Its position relates to the screen, not the
   reading direction.

2. **Decorative or absolute-positioned UI chrome.** A drop shadow offset,
   a notification badge pinned to the top-right of an avatar, a gradient
   angle — these are visual design decisions about the rendered surface,
   not about content flow.

3. **Fixed-direction graphics.** Arrows meaning "north/up" should not flip
   in RTL. Play/pause icons, volume sliders, timeline scrubbers — often
   physical, because the metaphor is a physical device, not a reading
   direction.

4. **Hardware-anchored interactions.** Drag handles on the right edge of a
   resizable panel, scrollbar styling, anything where pointer mechanics
   and muscle memory matter more than text flow.

5. **Print and physical-media styles.** Page margins for print often map
   to physical paper orientation, not content flow.

```css
/* Correct use of physical properties: viewport-anchored FAB */
.ds-fab {
  position: fixed;
  bottom: 24px;      /* physical: pinned to screen bottom */
  right: 24px;       /* physical: pinned to screen right */
}

/* Correct use of physical: shadow offset doesn't flip */
.ds-card {
  box-shadow: 2px 4px 8px rgba(0, 0, 0, 0.1);
}

/* Correct use of physical: notification badge position */
.ds-avatar__badge {
  position: absolute;
  top: -4px;         /* physical: visual placement on the avatar */
  right: -4px;       /* physical: stays top-right regardless of dir */
}
```

### The reference table

When a value *does* follow content flow, here are the logical equivalents:

| Physical                      | Logical                                                              |
| ----------------------------- | -------------------------------------------------------------------- |
| `margin-left`                 | `margin-inline-start`                                                |
| `margin-right`                | `margin-inline-end`                                                  |
| `margin-top`                  | `margin-block-start`                                                 |
| `margin-bottom`               | `margin-block-end`                                                   |
| `padding-left`                | `padding-inline-start`                                               |
| `padding-right`               | `padding-inline-end`                                                 |
| `border-left`                 | `border-inline-start`                                                |
| `border-right`                | `border-inline-end`                                                  |
| `left` (positioning)          | `inset-inline-start`                                                 |
| `right` (positioning)         | `inset-inline-end`                                                   |
| `top`                         | `inset-block-start`                                                  |
| `bottom`                      | `inset-block-end`                                                    |
| `text-align: left`            | `text-align: start`                                                  |
| `text-align: right`           | `text-align: end`                                                    |
| `float: left`                 | `float: inline-start`                                                |
| `width`                       | `inline-size`                                                        |
| `height`                      | `block-size`                                                         |
| `border-radius: 8px 0 0 8px` | `border-start-start-radius: 8px; border-end-start-radius: 8px;`     |

### Decision checklist for code review

When you see a physical property in component CSS, ask:

1. **Does this style relate to text, content, or a content container?**
   → Should be logical.
2. **Does this style relate to viewport position, device orientation,
   or hardware interaction?** → Physical is correct.
3. **Would I want this to flip in RTL or vertical writing modes?**
   → Logical if yes, physical if no.

When in doubt, the question "what does this value actually mean?" usually
resolves it. The same way you wouldn't use `<section>` and `<div>`
interchangeably because they render the same — logical and physical
properties mean different things, and using one to mean the other obscures
intent.

### Shorthand properties

CSS shorthand for margin and padding uses physical order (top, right,
bottom, left). For content-flow values, use logical shorthands:

```css
/* Physical shorthand — doesn't flip */
.ds-card { margin: 0 16px 0 0; }

/* Logical equivalent — flips inline-end in RTL */
.ds-card {
  margin-block: 0;
  margin-inline: 0 16px; /* start end */
}

/* Or use individual properties for clarity */
.ds-card {
  margin-inline-end: 16px;
}
```

## Text expansion and contraction

### The expansion problem

English is one of the most compact Western languages. Translations expand:

| Language   | Typical expansion from English |
| ---------- | ------------------------------ |
| German     | +30%                           |
| French     | +20%                           |
| Finnish    | +30–40%                        |
| Japanese   | -10–30% (fewer chars, wider)   |
| Arabic     | +25%                           |
| Chinese    | -50% character count, similar visual width |

A button labeled "Save" (4 chars) becomes "Speichern" (9 chars) in German
and "Tallenna" (8 chars) in Finnish.

### Design strategies for expansion

**Allow wrapping in labels and headings.** Don't set `white-space: nowrap`
on elements whose content you don't control:

```css
/* Bad: clips translated text */
.ds-button { white-space: nowrap; overflow: hidden; }

/* Good: allows wrapping when needed */
.ds-button {
  white-space: normal;
  /* Or: white-space: nowrap with overflow-wrap: break-word as fallback */
}
```

**Use min-width, not width.** Let elements grow:

```css
/* Bad: fixed width breaks with long translations */
.ds-button { width: 120px; }

/* Good: minimum width with growth allowed */
.ds-button { min-inline-size: 80px; }
```

**Test with pseudolocalization.** Before real translations exist, test
with artificially expanded strings:

```text
"Save" → "[Ṡààṿéé one two]" (doubled vowels + padding)
"Cancel" → "[Ćààñćééĺ one two three]"
```

This simulates expansion and reveals layout issues immediately.

### Truncation as last resort

When space is truly constrained (navigation items, table cells), truncation
is acceptable — but always with the full text accessible:

```html
<span class="ds-truncate" title="Benutzereinstellungen">
  Benutzerein…
</span>
```

Better: use a tooltip or expand-on-hover instead of relying on `title`
attribute alone (which doesn't work on touch devices and is slow to
appear).

## Number, date, and currency formatting

### Use Intl APIs, not hardcoded formats

The browser's `Intl` object handles locale-aware formatting. Components
that display dates, numbers, or currency should use it:

```javascript
// Number formatting
new Intl.NumberFormat('de-DE').format(1234567.89)
// → "1.234.567,89"

new Intl.NumberFormat('en-US').format(1234567.89)
// → "1,234,567.89"

// Date formatting
new Intl.DateTimeFormat('ja-JP', {
  year: 'numeric', month: 'long', day: 'numeric'
}).format(new Date(2025, 0, 15))
// → "2025年1月15日"

// Currency formatting
new Intl.NumberFormat('de-DE', {
  style: 'currency', currency: 'EUR'
}).format(1499.99)
// → "1.499,99 €"
```

### Component implications

**Date pickers:** don't hardcode MM/DD/YYYY. Accept a locale and derive
the format, first-day-of-week (Monday vs. Sunday vs. Saturday), month
names, and day abbreviations from `Intl.DateTimeFormat`.

**Number inputs / steppers:** the decimal separator is a comma in most of
Europe. Your component's parser needs to handle `1.234,56` and `1,234.56`
depending on locale.

**Data tables with numbers:** right-align number columns for readability.
In RTL layouts, this means the numbers are on the left side of the cell
— which is correct, because "end" alignment is what you want, and logical
properties handle it:

```css
.ds-table__cell--numeric {
  text-align: end;
  /* Also consider tabular-nums for alignment */
  font-variant-numeric: tabular-nums;
}
```

## String externalization in components

### Components should not hardcode user-facing strings

Every visible string in a component — button labels, ARIA labels, error
messages, placeholder text — must be overridable:

```html
<!-- Good: all strings are attributes/slots, with sensible defaults -->
<ds-pagination
  previous-label="Previous"
  next-label="Next"
  aria-label="Pagination"
>
</ds-pagination>
```

```javascript
// Web Component with i18n-friendly defaults
class DsPagination extends HTMLElement {
  get previousLabel() {
    return this.getAttribute('previous-label') ?? 'Previous';
  }
  get nextLabel() {
    return this.getAttribute('next-label') ?? 'Next';
  }
}
```

### ARIA labels especially

Screen reader users depend on ARIA labels being in their language. If your
component sets `aria-label="Close"` internally, that's English-only. Make
it configurable:

```html
<ds-dialog close-label="Schließen" aria-label="Dialogfenster">
```

### Pattern: i18n strings as a slot or config object

For components with many strings, accept a configuration object rather
than dozens of attributes:

```javascript
// Consumer provides translated strings
const i18n = {
  close: 'Schließen',
  next: 'Weiter',
  previous: 'Zurück',
  of: 'von',        // "3 von 10"
  results: 'Ergebnisse'
};

document.querySelector('ds-data-table').i18n = i18n;
```

## Font and script support

See `references/typography.md` for detailed font stack guidance. The key
i18n concerns:

### Line breaking

Different scripts have different line-breaking rules:
- **CJK:** can break between any two characters (no spaces between words)
- **Thai:** breaks at syllable boundaries (no spaces between words)
- **Arabic:** breaks between words, but cursive connections affect where
  breaks look clean

Use `overflow-wrap: break-word` as a safety net, but also consider
`word-break`:

```css
/* Safe for multi-script content */
.ds-prose {
  overflow-wrap: break-word;
  word-break: break-word; /* Deprecated but still useful in some engines */
}

/* For CJK-heavy contexts */
.ds-table__cell {
  word-break: break-all; /* Allow breaking within CJK words at any point */
}
```

### Vertical text

Some CJK contexts use vertical writing mode. If your system might encounter
this:

```css
.ds-vertical-text {
  writing-mode: vertical-rl;
  text-orientation: mixed; /* Latin chars rotated, CJK chars upright */
}
```

This is rare in web apps but relevant for publishing systems, signage, or
cultural content platforms.

## Input and form considerations

### Input methods

Different locales use different input methods (IME for CJK, diacritics for
European languages). Components must not interfere:

- **Don't intercept keydown events on text inputs** unless you're handling
  a specific key (Escape, Enter, ArrowDown). CJK IME composition fires
  multiple keydown events before committing a character. Intercepting them
  breaks input.

- **Use the `compositionstart` / `compositionend` events** to know when
  an IME is active, and suppress other keyboard handlers during composition:

```javascript
let isComposing = false;
input.addEventListener('compositionstart', () => { isComposing = true; });
input.addEventListener('compositionend', () => { isComposing = false; });

input.addEventListener('keydown', (e) => {
  if (isComposing) return; // Don't interfere with IME
  // Handle special keys
});
```

### Input type and inputmode

Set `inputmode` to hint at the right virtual keyboard:

```html
<input inputmode="numeric" pattern="[0-9]*">  <!-- Number pad -->
<input inputmode="email">                      <!-- @ key visible -->
<input inputmode="tel">                        <!-- Phone pad -->
<input inputmode="url">                        <!-- .com key, / key -->
```

Rely on the browser's native `<input>` types — they provide the right
keyboard for each locale automatically. Don't build a custom number
input just because you want a different visual style; the native behavior
is more work to replicate than you think.

### Placeholder text

Placeholder text is not a substitute for a label (WCAG 1.3.1, 3.3.2) and
has translation implications — it disappears on input, so translated
placeholder text that's too long won't be readable. Keep placeholders short
and supplement with persistent help text:

```html
<label for="search">Search</label>
<input id="search" placeholder="e.g. 'quarterly report'"
       aria-describedby="search-help">
<p id="search-help" class="ds-help-text">
  Search by title, author, or keyword.
</p>
```

## Testing for i18n readiness

### Automated checks

1. **Logical property lint:** use a CSS linter rule to flag physical
   properties (`left`, `right`, `margin-left`, etc.) in component CSS as
   review prompts — not errors. Physical properties are correct for
   viewport-anchored and chrome-related styles (see the logical properties
   section above). The lint rule should trigger a "is this intentionally
   physical?" review comment, not an automatic failure. Require a brief
   inline comment (e.g., `/* physical: viewport-anchored */`) to
   acknowledge intentional physical property usage and suppress the lint.

2. **Pseudo-localization:** run all component demos through a
   pseudolocalization transform that expands strings by 40% and adds
   accented characters. Any layout that breaks is a real i18n bug.

3. **Character coverage:** verify the font stack handles at least Latin,
   Cyrillic, Greek, Arabic, Hebrew, Devanagari, CJK, and Thai. Noto fonts
   as a fallback cover this.

### Manual checks

1. **Set `dir="rtl"` on the page** and verify every component mirrors
   correctly — padding, borders, icons, scroll direction, focus order.

2. **Paste long German/Finnish strings** into every label, button, and
   heading. Does it wrap or truncate gracefully?

3. **Paste Arabic or Hebrew text** into form inputs and editable content.
   Does caret positioning and selection work?

4. **Test with CJK input methods** (macOS: add Japanese or Chinese input
   source in System Preferences). Does the combobox, search input, and
   text field handle composition correctly?

5. **Test with a screen reader in a non-English locale** — VoiceOver in
   Spanish, NVDA in German — to verify ARIA labels are reaching the
   assistive technology correctly.
