# Microdata vs JSON-LD: format decision guide

Both formats are current, schema.org-supported ways to express the same structured data. Neither is deprecated. The choice is about where you want the risk to live, not about spec status.

## Current standards status

- HTML Microdata (the `itemscope`, `itemtype`, `itemprop`, `itemid`, `itemref` attributes) is defined in the WHATWG HTML Living Standard, [§5 Microdata](https://html.spec.whatwg.org/multipage/microdata.html). It is actively maintained there — this is the canonical spec to cite, not the old W3C draft.
- A separate W3C effort to standardize Microdata independently was discontinued in 2013 when the working group could not find an editor, and was published only as a Note rather than a Recommendation. That draft now redirects to the WHATWG page above. If you see "Microdata is deprecated" claims online, they're almost always describing that abandoned W3C fork, not the WHATWG syntax, which remains live.
- No current browser implements the old Microdata *DOM API* (`document.getItems()`, `HTMLPropertiesCollection`, etc.). That's irrelevant to this skill — the scripts here only ever write the attribute syntax, never rely on the JS API.
- JSON-LD is a W3C Recommendation ([JSON-LD 1.1](https://www.w3.org/TR/json-ld11/)) and is Google's most-documented format for structured data in its own developer guidance.

Neither format is more "correct" or more likely to be supported by search engines going forward — both are actively supported by Google, Bing, and schema.org tooling today.

## The actual tradeoff

**Microdata attaches structured data to the same DOM nodes a user sees.** This has one big advantage: the data literally cannot say something different from what's rendered, because it's the same element. The cost is that the markup becomes more verbose and more coupled to the template — if a component's HTML structure changes, the microdata attributes have to move with it, by hand or by whatever build step manages them.

**JSON-LD keeps structured data in a separate `<script type="application/ld+json">` block**, decoupled from the rendered DOM. This is easier to author, easier to validate (it's just JSON), and survives template refactors without anyone needing to touch it. The cost is exactly the flip side of Microdata's advantage: nothing enforces that the JSON-LD values match what's actually rendered. A price change in the visible markup that isn't mirrored in the JSON-LD block will silently misinform search engines — a real risk on any site where content and structured data are maintained by different people or different systems.

This is the same shape of problem as keeping an accessible name in sync with visible text: attaching the semantic data directly to the visible node (Microdata) guarantees sync at the cost of verbosity; maintaining it as a separate parallel representation (JSON-LD) is cleaner to write but requires discipline to keep aligned.

## Decision guide

Reach for **Microdata** when:
- You control the template tightly and it's stable (a versioned design-system component is the strongest case) — but if that component uses shadow DOM, see the Web Component caveat in `SKILL.md`: only Declarative Shadow DOM (server-rendered) keeps this reliable, and light-DOM/head placement is still the safer default.
- The cost of drift is high — pricing, availability, ratings, or anything where a mismatch between visible content and structured data could look like manipulation to a search engine or mislead a user.
- The site already has Microdata in place and the task is fixing/extending it, not migrating formats.

Reach for **JSON-LD** when:
- The markup is generated by a CMS, template engine, or multiple contributors, and keeping a separate script block in sync is more tractable than threading attributes through every possible template variant.
- The structured data includes properties that don't map cleanly onto any visible element (e.g., `aggregateRating.reviewCount` when only the average is shown on the page).
- You want the cleanest possible separation between presentation markup and machine-readable data for long-term maintainability.

When genuinely unsure, default to whichever technique from `SKILL.md` Step 1 the target already uses — a design-system component pushes toward Microdata; a CMS-driven static-site pipeline pushes toward JSON-LD generated at build time from the same content model that renders the page (which, notably, closes the drift risk for JSON-LD too, since both the visible markup and the JSON-LD are generated from one source of truth).
